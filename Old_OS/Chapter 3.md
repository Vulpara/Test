## 一、为什么选择ARM

**1. ARM 简介**

* ARM：一种**精简指令集**处理器架构。
* 核心优势：低功耗、低发热、高集成度。

**2. 发展趋势**

* 全球超过95%的智能手机和平板电脑（如iPhone、安卓手机）都采用ARM架构。
* 发展趋势：走向PC/服务器。

**3. ARMv8简介**

* 核心特色：安全隔离模块、小型独立的存储器、外设系统、随机数生成器、非易失性计数器
* 定义了两种执行状态，这使其既能处理64位任务，又能高效运行32位程序。
* 基本的冯诺依曼架构：计算、存储器（内存RAM）、输入输出（镜像加载、外设）

## 二、硬件执行逻辑

**1. 冯诺依曼架构**

* 输入/输出：和设备之间交互数据
* 中央处理器CPU：包括处理单元和控制单元（解析指令）
* 存储器：内存（存储指令和数据）

**2. 应用程序的硬件执行环境**

* 操作系统也是程序：大量的CPU指令、一些特权指令
* 指令集架构（Instruction Set Architecture, ISA）：与硬件绑定
	* 用户 ISA 举例
		* mov x0, sp
		* add x0, x0, #1
	* 系统ISA示例
		* msr vbar_el1, x0

**3. 源代码和机器指令的关系**

* 源代码：给人看的
* 机器码：给机器看的（汇编语言）

**4. 程序是怎么运行的**

* 程序启动，将机器码从硬盘中复制到内存中
* CPU 中的特殊寄存器（PC，Program Counter，程序计数器）记录下一条指令在内存中的位置
* 一个 C 语言程序从编写到被 CPU 执行的完整流程（如下图所示）：
	* 编译过程：源代码（人类可读的高级语言） → 汇编程序（符号化的机器语言）
	* 汇编过程：汇编程序 → 机器程序（机器码是 CPU 能直接识别的二进制指令）
	* 存储与加载：机器码 → 内存（当程序启动时，操作系统会把机器码加载到内存里）
	* CPU 执行阶段（PC 指向内存中第一条指令的地址）

## 三、ARM汇编语言

**1. ARM中常用的寄存器**

* 31个64位通用寄存器：X0～X30
	* 几个常用的寄存器：X0，X1，X8，X29，X30
	* 每个寄存器有多种形式，由系统模式决定
		* 64位形式：X0～X30，用Xn读写时，操作完整的64位
		* 32位形式：W0～W30，用Wn读写时，高32位自动清零
		* Wn就是Xn的低32位，共用同一个硬件寄存器，只是访问位数不同
* 32个128位额外寄存器：V0～V31
	* 一套和通用寄存器完全分开的额外寄存器，不占用通用寄存器的数量
	* 专门用于：浮点数计算、SIMD 操作、硬件加密操作
	* 同一个 128 位物理寄存器，可以用不同位宽的形式访问：
		* 128位形式：V0～V31，访问完整的128位数据
		* 64位形式：D0～D31，访问低64位数据
		* 32位形式：S0～S31，访问低32位数据
		* S0、D0、V0本质是同一个硬件寄存器的不同视角，和W0是X0低32位的逻辑完全一致
* 这样设计的好处：方便指令级并行

| 64位寄存器    | 32位寄存器    | 用途分类      | 具体用途                       |
| --------- | --------- | --------- | -------------------------- |
| X0        | W0        | 参数/返回值寄存器 | 函数返回值、第一个函数参数              |
| X1～X7     | W1～W7     | 参数寄存器     | 第二个～第八个函数参数                |
| X8        | W8        | 系统调用寄存器   | 存放系统调用的调用号（syscall number） |
| X9 ~ X15  | W9 ~ W15  | 临时寄存器     | 临时数据寄存器，调用者负责保存其值          |
| X16 ~ X18 | W16 ~ W18 | 平台相关寄存器   | 平台特定用途，一般由系统/编译器底层使用       |
| X19 ~ X28 | W19 ~ W28 | 被调用者保存寄存器 | 被调用函数若使用，需先保存原值、返回前恢复      |
| X29       | W29       | 帧指针（FP）   | 标记函数栈帧基地址，用于栈帧管理和调试        |
| X30       | W30       | 链接寄存器（LR） | 保存函数返回地址，配合`ret`指令实现函数返回   |

**3. CPU 视角下的内存**

* 寄存器中的数据（如函数参数 W0/W1、返回值）最终都来自内存，运算结果也需要写回内存，CPU 无法直接操作内存，必须通过专用的内存存取指令完成
* 存取指令要实现精准读写内存某一位置，就需要寻址方式（定义如何计算内存有效地址）

**4. 寻址方式**

* 基址寻址（Base Register Addressing）
	* ARM64 最基础、最核心的寻址方式，无任何偏移，直接以寄存器值作为内存地址
	* 核心原理：
		* 选定一个通用寄存器 Xn 作为基址寄存器，其内部存储的数值 = 内存有效地址
		* 直接访问该有效地址的内存单元，完成数据读写
		* 基址寄存器 Xn 的值不会被修改
	* 适用于访问单个固定内存地址
* 变址寻址（Offset Addressing）
	* 基址寻址的基础扩展版，增加立即数偏移，解决连续内存地址访问问题，是 ARM64 最常用的寻址方式
	* 核心原理：
		* 选定一个通用寄存器 Xn 作为基址寄存器，有效地址 = 基址寄存器值 + 立即数偏移
		* 访问该有效地址的内存单元，完成数据读写
		* 基址寄存器 Xn 的值仍不会被修改（仅计算临时有效地址）
	* 适用于访问连续固定内存地址
* 前变址寻址（Pre-index Addressing）
	* 变址寻址的进阶版，先更新基址，再访问内存，带基址寄存器自更新
	* 核心原理：
		* 计算临时有效地址 = 基址寄存器值 + 立即数偏移
		* 先更新基址寄存器：将临时有效地址赋值给 Xn（永久修改）
		* 用更新后的有效地址访问内存单元，完成数据读写
	* 适用于循环访问连续内存
* 后变址寻址（Post-index Addressing）
	* 变址寻址的另一进阶版，先访问内存，再更新基址，同样带基址寄存器自更新
	* 核心原理：
		* 先以基址寄存器原始值作为有效地址，访问内存单元，完成数据读写
		* 计算新地址 = 基址寄存器原始值 + 立即数偏移
		* 后更新基址寄存器：将新地址赋值给 Xn（永久修改）
	* 适用于循环访问连续内存

**5. ARM64 条件执行的逻辑**

* 类型 1：根据本条指令的状态直接判断
* 类型 2：根据前一条指令的状态间接判断
* CPU 中有一个特殊的状态寄存器 PSTATE，其中专门划分了一个条件码域，用于存储运算指令执行后的状态信息（即条件码）。所有依赖前一条指令的条件跳转指令，都会从 PSTATE 中读取条件码，作为跳转判断的依据
* PSTATE 中最核心的4个条件码标志位

|标志位|英文全称|含义|核心判断场景|
|---|---|---|---|
|N|Negative|负标志|运算结果是否为负数（有符号数）|
|Z|Zero|零标志|运算结果是否为 0|
|C|Carry|进 / 借位标志|无符号数运算是否产生进位（加法）/ 借位（减法）|
|V|Overflow|溢出标志|有符号数运算是否产生溢出|

* 最常用的标志位：**Z（零标志）**—— `bne`就是基于`Z=0`（结果非零）判断，是循环、分支中最常用的条件。

**6. 函数的调用、返回与栈**

* 函数有局部变量需要保存，特别是递归函数
* 每个函数拥有的连续内存空间称为函数的栈帧（Stack Frame），栈帧在内存中的起始位置称为帧指针（Frame Pointer，FP），存于x29 通用寄存器
* 运行时栈（Runtime Stack）：内存中一块连续的、后进先出（LIFO）的存储区域，函数调用时，保存当前函数的所有临时 / 局部状态
* 三个核心寄存器：
	* SP（Stack Pointer），栈指针：永远指向栈顶，入栈/出栈时自动移动
	* FP（X29），帧指针：永远指向当前栈帧的底部（固定不变，用来定位局部变量）
	* LR（X30），链接寄存器：保存函数返回地址，`ret` 指令靠它跳回调用方
* 完整模拟：案例代码C语言对应的ARM64汇编语言

```C
// 被调用函数：有2个局部变量 
int func(int a, int b) 
{ 
    int x = 10; // 局部变量1 
    int y = 20; // 局部变量2 
    return a + b + x + y; 
} 
// 调用函数 
int main() 
{ 
    func(1, 2); // 调用func return 0; 
}
```

对应的 ARM64 汇编语言：

```asm
// ========== func 函数 ========== 
func: 
    // 1. 创建栈帧：入栈保存 FP/LR（关键！保存上一层状态） 
    stp x29, x30, [sp, #-16]! // 前变址寻址：SP -=16，存储FP、LR到栈，保存上一层函数的现场
    mov x29, sp               // FP = SP，设置当前栈帧基址 
    
    // 2. 分配局部变量空间（栈再向下扩展16字节，存x、y） 
    sub sp, sp, #16 
    
    // 3. 初始化局部变量 
    mov w9, #10 
    str w9, [x29, #-4]  // x=10 → 存到 FP-4 的位置 
    mov w10, #20 
    str w10, [x29, #-8] // y=20 → 存到 FP-8 的位置 
    
    // 4. 计算结果 
    ldr w9, [x29, #-4] 
    ldr w10, [x29, #-8] 
    add w0, w0, w1 
    add w0, w0, w9 
    add w0, w0, w10 
    
    // 5. 销毁栈帧：恢复SP/FP，出栈LR 
    mov sp, x29             // 恢复SP到栈帧底部 
    ldp x29, x30, [sp], #16 // 后变址寻址：恢复FP、LR，SP +=16 
    ret                     // 跳回LR地址 

// ========== main 函数 ========== 
main: 
    stp x29, x30, [sp, #-16]! 
    mov x29, sp 
    mov w0, #1  // 参数1 
    mov w1, #2  // 参数2 
    bl func     // 调用func：LR = 返回地址 
    mov w0, #0 
    ldp x29, x30, [sp], #16 
    ret
```

## 四、内存模型

**1. 冯诺依曼架构的前提**

* 数据和程序都存在同一片内存中，共享同一个地址空间
* 指令、数据、外设寄存器，都用内存地址来访问

**2. 三个核心问题与OS的解决方案**

* 权限管理：内核是系统的大脑，如果应用程序能随便修改内核数据，系统会直接崩溃。
	* **CPU 特权级**：把 CPU 分成**内核态**和**用户态**，只有内核态才能访问内核内存。
	* **内存保护（页表权限）**：给每一段内存标记权限（读/写/执行），比如内核代码段设为只读 + 可执行，应用无法修改；应用内存段只能被自己进程访问。
* 适应不同内存空间：不同设备物理内存差异极大，应用不能直接用物理地址，否则换个设备就跑不了。
	* 虚拟内存：给每个进程提供独立的虚拟地址空间。
	* OS负责把虚拟地址映射到真实物理内存，物理内存不够时还能换页到磁盘，让应用感知不到底层差异。
* 统一管理所有外设：外设（键盘、硬盘、网卡、屏幕等）硬件接口千差万别，CPU 不能为每种外设做一套指令。
	* 内存映射 I/O：把外设的寄存器、缓冲区映射到内存地址空间里，让 CPU 访问外设就像访问普通内存一样。

## 五、特权级模型

**1. 特权级模型**

* 4个异常级别（Exception Level，EL），2个和安全相关的
* EL0异常级别最低，EL3异常级别最高
* 如果不开启安全模式和虚拟化，就不用管EL2和EL3

![](计算机/操作系统/attachments/C3-4.png)

**2. ARM中常用的寄存器**

* 31个64位通用寄存器
* 1个PC寄存器
* 4个栈寄存器（切换时保存SP）：SP_EL0，SP_EL1，SP_EL2，SP_EL3
* 3个异常链接寄存器（保存异常的返回地址）：ELR_EL1，ELR_EL2，ELR_EL3
* 3个程序状态寄存器（切换时保存PSTATE）：SPSR_EL1，SPSR_EL2，SPSR_EL3

**3. 处理器何时会从 EL0 进入 EL1 ？**

* 执行系统调用，具体指令为特权调用：svc（supervisor call）
* 应用执行的指令触发了异常
* CPU 收到了外设发来的中断信息

## 六、设备与中断

**1. CPU如何得知键盘输入**

* 键盘等外设具有控制器和缓冲区，将输入存入缓冲区
* CPU获取该输入的可能方法：
	* 轮询：CPU不断去读该缓冲区中的值
	* 中断：当控制器接收到输入后，打断CPU正常执行，OS进行处理

**2. 通用概念**

* 中断（Interrupt）：
	* 外部硬件设备所产生的信号
	* 异步：产生原因和当前执行指令无关，如程序被磁盘读打断
* 异常（Exception）：
	* 软件的程序执行而产生的事件
	* 包括系统调用（System Call）
	* 用户程序请求操作系统提供服务
	* 同步：产生和当前执行或试图执行的指令相关

| 通用概念 | 产生原因 | AArch64       | x86-64               |     |
| ---- | ---- | ------------- | -------------------- | --- |
| 中断   | 硬件异步 | 异步异常（重置/中断）   | 中断（可屏蔽/不可屏蔽）         |     |
| 异常   | 软件同步 | 同步异常（终止/异常指令） | 异常（Fault/Trap/Abort） |     |

**3. AArch64 的中断（异步异常）**

* 重置（Reset）
	* 最高级别的异常，用以执行代码初始化CPU核心
	* 由系统首次上电或控制软件、看门狗等触发
* 中断（Interrupt）
	* CPU外部的信号触发，打断当前执行
	* 如计时器中断、键盘中断等

**4. AArch64 的异常（同步异常）**

* 中止（Abort）
	* 失败的指令获取或数据访问
	* 如访问不可读的内存地址等
* 异常产生指令（Exception generating instructions）
	* SVC（Supervisor Call）：用户程序-> 操作系统
	* HVC（Hypervisor Call）：客户系统-> 虚拟机管理器
	* SMC（Secure Mode Call）：Normal World -> Secure World

**5. x86-64 术语**

* 中断（设备产生、异步）
	* 可屏蔽：设备产生的信号，通过中断控制器与处理器相连，可被暂时屏蔽（如，键盘、网络事件）
	* 不可屏蔽：一些关键硬件的崩溃（如，内存校验错误）
* 异常（软件产生、同步）
	* 错误（Fault）: 如缺页异常（可恢复）、段错误（不可恢复）等
	* 陷阱（Trap）: 无需恢复，如断点（int 3）、系统调用（int 80）
	* 中止（Abort）: 严重的错误，不可恢复（机器检查）

**6. AArch64中断的分类**

* IRQ（Interrupt Request）：普通中断，优先级低，处理慢
* FIQ（Fast Interrupt Request）：一次只能有一个FIQ，快速中断，优先级高，处理快，常为可信任的中断源预留
* SError（System Error）：原因难以定位、较难处理的异常，多由异步中止（Abort）导致，如从缓存行（Cacheline）写回至内存时发生的异常

**7. 中断产生的中间环节：中断控制器的处理**

* 通用中断控制器（Generic Interrupt Controller，GIC）是中断产生的核心硬件，所有外设的中断信号都需先传入 GIC，经处理后再发送给 CPU
* GIC 的核心工作包括 4 点，也是中断信号在 GIC 中的产生与处理流程：
	* 接收外设原始中断信号：GIC 会为每个外设分配对应的信号接口，接收外设发送的中断信号，同时标记该中断的状态
		* Inactive：中断源未触发
		* Pending：中断源已触发，CPU 尚未处理
		* Active：CPU 正在处理该中断
		* Active and pending：同一中断源的一个实例正在处理，下一个实例已触发并等待
	* 中断的物理触发方式（外设→GIC 的信号传递形式）
		* 电平触发：外设的中断信号以**电位高低**作为触发标志，只要电位保持触发状态，GIC 会持续识别该中断；状态转换为`Inactive→Pending→Active and Pending→Active`
		* **边沿触发**：外设的中断信号以**电位跳变（上升 / 下降沿）** 作为触发标志，仅在跳变瞬间触发，后续电位恢复也不影响；状态转换为`Inactive→Pending→Active→Pending`
	* 中断的优先级管理与嵌套：GIC 会为不同中断分配优先级（如计时器中断优先级高于键盘），若低优先级中断正在处理时，高优先级中断产生，GIC 会触发嵌套中断，优先将高优先级中断信号发送给 CPU，这是中断产生过程中 GIC 的核心调度逻辑
	* 中断的路由与分发：GIC 的核心功能是分发：管理所有中断源，根据系统配置，将中断信号路由到指定的 CPU 核（多核心场景），并为每个 CPU 核提供独立的接口，确保中断信号准确传递到目标处理单元

**8. 中断产生的最后一步：GIC→CPU 的信号传递**

* 经 GIC 处理（优先级、路由、状态标记）后的中断信号，会被发送到指定的 CPU 核，CPU 接收到 IRQ/FIQ/SError 信号后，会立刻打断当前的正常执行流程，并触发特权级切换（若当前在 EL0 用户态，会切换到 EL1 内核态），至此，中断正式产生，后续进入 CPU 的异常处理阶段
* 中断产生的完整流程：
	* 外设触发条件满足（如键盘输入）
	* 外设产生原始中断信号
	* GIC 接收信号，标记状态、判断优先级、路由分发
	* GIC 将处理后的中断信号发送到指定 CPU 核
	* CPU 接收信号，打断当前执行、切换特权级
	* 中断正式产生，进入处理阶段

**9. 设备的内存映射**

* 设备内存映射的核心背景与目的：CPU 无法直接操作外设的硬件寄存器 / 缓冲区，为了让 CPU 能像访问内存一样和外设交互，系统将外设的硬件寄存器、数据缓冲区映射到物理内存的固定地址段
* 实现机制：内存映射 I/O（Memory-Mapped I/O，MMIO）：MMIO 复用 CPU 访问内存的`ldr`（读）、`str`（写）指令，替代专用的 I/O 指令，让 CPU 通过访问这些特殊的内存地址，等价于操作外设的寄存器，硬件层面会自动将对这些地址的访问转发到对应的外设控制器，而非实际的内存单元
* ARMv8 的虚拟地址空间统一划分，保证了访问安全和寻址统一
* 外设的映射地址属于内核地址空间，只有高特权级（EL1 及以上）能访问，保证了用户态无法直接操作外设，实现了硬件层面的隔离
* 设备内存映射的扩展应用：配置中断控制器（GIC）
	* 启用计时器中断
	* 获取 GIC 中断信息
* MMIO 的优势：访问方式统一、特权级隔离、硬件交互高效、全硬件覆盖、与中断联动

**10. 中断和异常处理的通用前提**

* 中断（硬件异步）和异常（软件同步）共用一套硬件处理机制，二者的唯一差异仅体现在最终选择的处理函数（handler）不同，这是所有架构异常处理的基础原则
* 处理的核心目标一致：在不破坏原有执行流的前提下，切换到高特权级完成事件处理，处理后精准切回低特权级继续执行
* 中断和异常处理是的通用事项：
	* 进入中断 / 异常时
		* 保存处理器当前状态（PSTATE），为后续恢复执行流做准备
		* 准备高特权级（如 AArch64 的 EL1）的执行环境（如切换内核栈）
		* 根据中断 / 异常类型，选择对应的处理函数（handler）
		* 严格保证用户态（EL0）和内核态（EL1）的隔离，禁止用户态直接访问内核资源
	* 处理过程中
		* 精准获取异常相关信息，如系统调用的参数、中断的触发源、错误的产生原因（如内存访问越界），为处理逻辑提供依据
	* 处理完成返回时
		* 恢复处理器之前保存的状态，还原到中断 / 异常发生前的执行环境
		* 从高特权级切回低特权级（如 EL1→EL0）
		* 恢复原有执行流，继续执行中断 / 异常发生前的指令

**11. AArch64 架构的异常处理**

* AArch64 架构异常处理依赖**专用系统寄存器 + 异常向量表 + 特权级自动切换**实现，流程高度自动化，硬件承担了大部分上下文管理工作
* 步骤 1：异常发生，硬件自动保存上下文与异常信息
	* 当 CPU 在 EL0（用户态）发生中断 / 异常时，硬件会自动将关键信息存入 EL1 专属的系统寄存器（这些寄存器禁止 EL0 访问，保证安全），无需软件干预，核心寄存器的作用如下：

| 寄存器      | 存储的信息                  | 作用              |
| -------- | ---------------------- | --------------- |
| SPSR_EL1 | 处理器当前状态（PSTATE）        | 为返回时恢复状态做准备     |
| ELR_EL1  | 当前程序计数器（PC）值           | 记录中断 / 异常发生的位置  |
| ESR_EL1  | 异常产生的原因（同步异常 / SError） | 如系统调用 ID、内存访问错误 |
| GIC 寄存器  | 中断触发源信息（异步中断）          | 如计时器 / 键盘中断的编号  |
* 步骤 2：硬件自动切换到 EL1 内核态，并切换栈指针
	* 硬件会修改处理器状态（PSTATE），强制从 EL0 切换到 EL1，进入高特权级执行处理逻辑
	* 栈指针自动从用户栈（SP_EL0）切换为内核栈（SP_EL1），实现用户态和内核态的栈隔离，避免用户态程序破坏内核栈数据
	* 特殊需求下，可通过配置 SPSel 寄存器，让 EL1 继续使用 SP_EL0 作为栈指针
* 步骤 3：通过异常向量表寻找对应的处理函数（handler）
	* 向量表的基址：存储在 VBAR_EL1 寄存器中，内核初始化时会通过代码将向量表地址写入该寄存器
	* 向量表的表项划分：表项按3 个维度划分，每个表项对应一段跳转汇编代码（128 字节对齐），CPU 根据中断 / 异常的实际情况自动选择表项，划分维度为：
		* 异常类型：同步异常、IRQ（普通中断）、FIQ（快速中断）、SError（系统错误）
		* 异常发生的特权级：EL0/EL1
		* 处理器运行状态：AArch64/AArch32（仅 EL0）
	* 典型表项偏移：如EL0 在 AArch64 状态下的同步异常，会跳转到`VBAR_EL1 + 0x400`的表项，这也是系统调用（SVC）的核心跳转入口
* 步骤 4：系统调用（SVC） ：异常向量表的实际应用
* 步骤 5：处理完成，通过 eret 指令实现异常返回
	* AArch64 通过专用的 eret（Exception Return）指令完成返回，硬件会自动执行以下操作，实现 EL1→EL0 的无缝切回：
		* 将 ELR_EL1 中保存的 PC 值还原到处理器的 PC 寄存器，定位到中断 / 异常发生前的下一条指令
		* 将 SPSR_EL1 中保存的 PSTATE 值还原到处理器，恢复原有执行状态
		* 栈指针自动从 SP_EL1 切回 SP_EL0，回到用户栈
		* 处理器在 EL0 继续执行原有程序流，完成整个异常处理流程

**12. x86-64 架构的异常处理（与 AArch64 的核心差异）**

| 对比维度  | AArch64 架构         | x86-64 架构                   |
| ----- | ------------------ | --------------------------- |
| 上下文保存 | 硬件自动存入 EL1 专用系统寄存器 | 硬件自动存入**内核栈**               |
| 异常向量表 | 按特权级 / 状态分级，存跳转代码  | 不分级，直接存储 handler 的地址        |
| 返回指令  | 专用 eret 指令，硬件自动恢复  | 专用 iret 指令，从内核栈恢复上下文        |
| 特权级切换 | EL0→EL1，硬件自动完成     | 用户态→内核态（Ring3→Ring0），硬件自动完成 |

**13. 中断处理的工程优化**

* 中断处理不能做太多事情，原因是中断处理过程中 CPU 会屏蔽低优先级中断（部分架构会屏蔽所有中断），若处理逻辑过久，会导致其他中断被阻塞，降低系统的实时性和响应速度
* Linux 系统提出了中断处理拆分策略：Top half + Bottom half
	* Top half（顶半部）：硬中断处理，快速完成
		* 属于强制的中断处理逻辑，在关中断状态下执行，要求代码极简、执行时间极短
		* 核心工作：确认中断触发源、标记中断事件（如将数据放入缓冲区）、清除外设的中断标志、开启中断，完成后立即返回，尽可能减少中断阻塞时间
	* Bottom half（底半部）：软中断处理，延迟执行
		* 属于非强制的处理逻辑，在开中断状态下执行，可在系统空闲时调度运行，不阻塞硬件中断
		* 核心工作：处理中断的非关键逻辑，如数据解析、数据拷贝、业务处理等
		* 实现方式：softirq（软中断）、tasklets（小任务）、工作队列、内核线程（PPT 明确这四类是底半部的实现形式）
* Top half 保证 “中断被快速响应”，Bottom half 保证 “中断被完整处理”，二者结合解决了 “中断响应速度” 和 “处理复杂性” 的矛盾，这也是 Linux 系统高实时性的重要原因

**14. 异常处理的核心设计目标**

* 安全性：通过特权级自动切换、专用寄存器 / 内核栈隔离、用户态禁止访问内核资源，保证用户态程序无法破坏内核和硬件
* 高效性：硬件自动完成上下文保存、特权级切换、向量表跳转，减少软件干预，让中断 / 异常能被快速响应和处理
* 实时性：通过 Linux 的 Top half & Bottom half 拆分策略，避免中断处理阻塞其他硬件事件，保证系统对外部设备的实时响应
